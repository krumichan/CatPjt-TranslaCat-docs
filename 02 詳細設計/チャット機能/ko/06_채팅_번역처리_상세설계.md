# 채팅 번역처리 상세설계

## 1. 문서 목적

본 문서는 TranslaCat 채팅 기능 1차 MVP에서 메시지 번역을 비동기로 처리하는 상세 흐름을 정의한다.

1차 목표는 메시지 원문 송수신을 막지 않고, 번역 결과를 나중에 받아 화면에 반영하는 것이다.

---

## 2. 기본 원칙

- 원문 메시지는 즉시 저장하고 즉시 WebSocket으로 발행한다.
- 번역은 비동기로 처리한다.
- 번역 실패가 메시지 송수신을 막지 않는다.
- 번역 상태는 DB에 저장한다.
- 번역 완료/실패는 WebSocket 이벤트로 전달한다.

---

## 3. 처리 흐름

```text
1. 사용자가 메시지 전송
2. BE가 채팅방 멤버 권한 검증
3. BE가 chat_message에 원문 저장
4. BE가 번역 대상 언어 계산
5. BE가 chat_message_translation PENDING 생성
6. BE가 chat.message.created 발행
7. BE가 chat.translation.pending 발행
8. BE가 AI Server에 번역 요청
9. AI Server가 번역 결과 반환
10. BE가 번역 결과 저장
11. BE가 chat.translation.completed 발행
```

실패 시:

```text
AI Server 요청 실패 또는 번역 실패
→ chat_message_translation.status = FAILED
→ failure_reason 저장
→ chat.translation.failed 발행
```

---

## 4. 번역 대상 언어 계산

초기 정책:

```text
채팅방 active member의 translationLanguageCode 수집
발신자 제외
중복 제거
null 제거
```

예시:

| 사용자 | 역할 | 번역 언어 |
|---|---|---|
| A | 발신자 | ko |
| B | 수신자 | ja |
| C | 수신자 | en |
| D | 수신자 | ja |

대상 언어:

```text
ja, en
```

---

## 5. chat_message_translation 상태

```text
PENDING
COMPLETED
FAILED
```

### PENDING

번역 요청 전 또는 요청 중 상태.

### COMPLETED

번역 결과가 정상 저장된 상태.

### FAILED

번역 요청 또는 결과 처리에 실패한 상태.

---

## 6. AI Server 요청 후보

```text
POST /api/v1/chat/translate
```

Request:

```json
{
  "sourceLanguageCode": "ko",
  "targetLanguageCodes": ["ja", "en"],
  "text": "오늘 뭐 먹을래?",
  "context": "chat"
}
```

Response:

```json
{
  "translations": [
    {
      "languageCode": "ja",
      "text": "今日何食べる？"
    },
    {
      "languageCode": "en",
      "text": "What do you want to eat today?"
    }
  ]
}
```

---

## 7. prompt 기준

AI Server의 채팅 번역 prompt는 다음 기준을 따른다.

- 짧은 채팅체를 유지한다.
- 원문 의미를 바꾸지 않는다.
- 이모지를 유지한다.
- URL을 유지한다.
- `@mention`을 유지한다.
- 금액, 날짜, 시간, 고유명사를 보존한다.
- 설명을 붙이지 않는다.
- 번역문만 반환한다.
- 욕설/비속어는 임의로 순화하지 않는다. 단, 정책상 필요한 경우 별도 검토한다.

---

## 8. BE 비동기 처리 방식

1차 후보:

```text
Spring @Async
ApplicationEvent + @TransactionalEventListener(AFTER_COMMIT)
```

권장 흐름:

```text
메시지 저장 transaction commit
→ AFTER_COMMIT event
→ 번역 비동기 요청
→ 번역 결과 저장 transaction
→ WebSocket 번역 완료 이벤트 발행
```

Redis Queue 또는 외부 message queue는 1차 범위에서 제외한다.

---

## 9. 실패 처리

실패 유형 후보:

```text
AI_SERVER_TIMEOUT
AI_SERVER_ERROR
INVALID_TRANSLATION_RESPONSE
TRANSLATION_EMPTY
UNKNOWN_ERROR
```

처리 기준:

- 실패한 언어만 FAILED 처리한다.
- 일부 언어 성공, 일부 언어 실패를 허용한다.
- 실패 시 원문 메시지는 유지한다.
- FE는 실패 상태를 표시하거나 조용히 숨길 수 있다.

---

## 10. 재처리 정책

1차에서는 자동 재처리를 필수로 구현하지 않는다.

후속 후보:

- 관리자 수동 재처리
- 실패 번역 자동 재시도
- 특정 메시지 번역 재요청 API
- 번역 실패 모니터링

---

## 11. 1차 완료 기준

- 메시지 저장 후 PENDING 번역 row가 생성된다.
- AI Server 번역 요청이 비동기로 수행된다.
- 번역 성공 시 COMPLETED로 저장된다.
- 번역 실패 시 FAILED로 저장된다.
- 번역 완료 이벤트가 WebSocket으로 발행된다.
- 번역 실패 이벤트가 WebSocket으로 발행된다.
- 번역 실패가 원문 메시지 송수신을 막지 않는다.
