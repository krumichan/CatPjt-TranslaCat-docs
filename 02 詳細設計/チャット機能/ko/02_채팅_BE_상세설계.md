# 채팅 BE 상세설계

## 1. 문서 목적

본 문서는 TranslaCat 채팅 기능 1차 MVP 구현을 위한 Backend 상세설계를 정의한다.

1차 MVP에서는 다음 흐름을 우선 구현한다.

- 채팅방 생성
- 채팅방 목록/상세 조회
- 채팅방 멤버 및 언어 설정 관리
- 메시지 저장/조회
- WebSocket 기반 실시간 메시지 송수신
- 메시지 원문 즉시 전송
- 메시지 번역 비동기 처리
- 번역 완료/실패 이벤트 전송

2차 이후 확장 기능인 오픈 채팅, AI 채팅 멤버, 읽음/미읽음, 온라인 상태, 알림, Redis, 가계부 연동은 1차 구현 범위에서 제외한다. 단, 향후 확장을 방해하지 않도록 도메인 결합도를 낮게 유지한다.

---

## 2. 설계 원칙

### 2.1 1차 MVP 우선 원칙

1차 개발에서는 채팅 기능의 핵심 흐름을 검증하는 데 집중한다.

```text
사용자 메시지 전송
→ 원문 메시지 DB 저장
→ WebSocket으로 원문 즉시 발행
→ 번역 대기 상태 생성
→ AI Server 비동기 번역 요청
→ 번역 결과 저장
→ WebSocket으로 번역 완료 이벤트 발행
```

아래 기능은 1차에서 구현하지 않는다.

- 오픈 채팅방 검색/참여
- AI 채팅 멤버 응답
- 읽음/미읽음
- 온라인 상태
- 채팅 알림
- Redis 기반 scale-out
- 채팅방-가계부 연동

---

### 2.2 계층 분리 원칙

채팅 도메인은 단순 CRUD보다 흐름이 길기 때문에 Controller-Service-Repository 구조만 사용하지 않는다.
기능 복잡도에 따라 다음 계층을 사용한다.

```text
Controller / WebSocketController
  ↓
Facade
  ↓
CommandService / QueryService / DomainService
  ↓
Repository / QueryRepository / External Client
  ↓
Event Publisher
```

각 계층의 역할은 다음과 같다.

| 계층 | 역할 |
|---|---|
| Controller | HTTP 요청 수신, DTO 검증, 인증 사용자 전달 |
| WebSocketController | STOMP 메시지 수신, 메시지 전송 요청 처리 |
| Facade | 하나의 사용자 use case 조립, 여러 service 호출 |
| CommandService | 생성/수정/삭제 등 DB 변경 처리 |
| QueryService | 조회 전용 처리, QueryDSL 기반 조회 호출 |
| DomainService | 권한, 정책, 번역 대상 계산 등 도메인 규칙 처리 |
| Repository | JPA 기본 persistence 처리 |
| QueryRepository | QueryDSL 기반 복잡한 조회 처리 |
| External Client | AI Server 등 외부 API 호출 |
| Event Publisher | WebSocket 이벤트 발행 |

## 3. 패키지 구조 후보

기존 TranslaCat BE 구조에 맞춰 `domain` 하위에 `chat` 도메인을 추가한다.

```text
jp.co.translacat.domain.chat
├─ room
│  ├─ controller
│  ├─ dto
│  ├─ entity
│  ├─ enums
│  ├─ facade
│  ├─ repository
│  └─ service
├─ member
│  ├─ dto
│  ├─ entity
│  ├─ enums
│  ├─ repository
│  └─ service
├─ message
│  ├─ controller
│  ├─ dto
│  ├─ entity
│  ├─ enums
│  ├─ facade
│  ├─ repository
│  └─ service
├─ translation
│  ├─ dto
│  ├─ entity
│  ├─ enums
│  ├─ repository
│  └─ service
├─ websocket
│  ├─ controller
│  ├─ dto
│  ├─ event
│  ├─ publisher
│  └─ config
└─ client
   └─ ai
```

초기 구현에서는 패키지를 과도하게 세분화하지 않아도 된다. 다만 다음 구분은 유지한다.

- 채팅방 관련: `room`
- 멤버/권한/언어 설정: `member`
- 메시지 원문: `message`
- 메시지 번역: `translation`
- WebSocket 이벤트: `websocket`
- AI Server 호출: `client.ai`

## 4. 주요 Entity

### 4.1 ChatRoom

채팅방 기본 정보를 관리한다.

주요 필드 후보:

```text
id
roomType
name
description
ownerId
active
createdAt
updatedAt
deletedAt
```

1차 MVP의 `roomType`은 다음을 사용한다.

```text
DIRECT
GROUP
```

2차 이후 확장을 위해 enum에는 `OPEN` 추가를 고려한다. 단, 1차 API에서는 OPEN 생성/조회 기능을 제공하지 않는다.

---

### 4.2 ChatRoomMember

채팅방 참여자와 채팅방별 언어 설정을 관리한다.

주요 필드 후보:

```text
id
chatRoom
user
role
originalLanguageCode
translationLanguageCode
showOriginal
showTranslation
active
joinedAt
leftAt
createdAt
updatedAt
deletedAt
```

1차 MVP의 role 후보:

```text
OWNER
ADMIN
MEMBER
```

1차에서는 OWNER/MEMBER 중심으로 구현하고, ADMIN은 2차 이후 확장으로 사용해도 된다.

---

### 4.3 ChatMessage

채팅 메시지 원문을 관리한다.

주요 필드 후보:

```text
id
chatRoom
senderUser
senderType
messageType
content
status
createdAt
updatedAt
deletedAt
```

1차 MVP의 `senderType`은 다음을 사용한다.

```text
USER
```

2차 AI 채팅 멤버 확장을 위해 enum에는 `AI` 추가를 고려한다.

1차 MVP의 `messageType`은 다음을 사용한다.

```text
TEXT
SYSTEM
```

1차 메시지 전송은 `TEXT` 중심으로 구현한다.

---

### 4.4 ChatMessageTranslation

메시지별 번역 결과와 상태를 관리한다.

주요 필드 후보:

```text

chatMessage
languageCode
translatedContent
status
failureReason
createdAt
updatedAt
completedAt
```

translation status 후보:

```text
PENDING
COMPLETED
FAILED
```

unique 제약 후보:

```text
message_id + language_code
```

## 6. 주요 Service 설계

### 6.1 ChatRoomFacade

채팅방 생성과 관련된 use case를 조립한다.

주요 메서드 후보:

```java
ChatRoomResponseDto createRoom(Long loginUserId, ChatRoomCreateRequestDto request)
ChatRoomDetailResponseDto getRoomDetail(Long loginUserId, Long roomId)
```

역할:

- 채팅방 생성 흐름 조립
- 멤버 생성 service 호출
- 1:1 채팅방 중복 생성 정책 검토
- 생성자 OWNER 등록

---

### 6.2 ChatRoomCommandService

채팅방 생성/수정/삭제 등 변경 작업을 담당한다.

주요 메서드 후보:

```java
ChatRoom createRoom(ChatRoomType roomType, String name, String description, Long ownerId)
void softDeleteRoom(ChatRoom room)
```

---

### 6.3 ChatRoomQueryService

채팅방 목록/상세 조회를 담당한다.

주요 메서드 후보:

```java
List<ChatRoomListResponseDto> getMyRooms(Long loginUserId)
ChatRoomDetailResponseDto getRoomDetail(Long loginUserId, Long roomId)
```

QueryDSL 사용 후보:

- 내 채팅방 목록 조회
- 마지막 메시지 포함 조회
- 멤버 수 포함 조회
- 검색 조건 추가 시 동적 쿼리

---

### 6.4 ChatRoomAccessService

채팅방 접근 권한을 검증한다.

주요 메서드 후보:

```java
ChatRoomMember validateRoomMember(Long roomId, Long userId)
void validateCanSendMessage(Long roomId, Long userId)
void validateCanManageRoom(Long roomId, Long userId)
```

검증 기준:

- 채팅방이 존재하는가
- 채팅방이 삭제되지 않았는가
- 사용자가 active member인가
- 메시지 전송 가능한 role인가

---

### 6.5 ChatRoomMemberCommandService

채팅방 멤버 생성/퇴실/언어 설정 변경을 담당한다.

주요 메서드 후보:

```java
ChatRoomMember addMember(ChatRoom room, User user, ChatRoomMemberRole role)
void leaveRoom(Long roomId, Long userId)
void updateLanguageSettings(Long roomId, Long userId, ChatLanguageSettingRequestDto request)
```

---

### 6.6 ChatRoomMemberQueryService

멤버 목록과 멤버별 언어 설정 조회를 담당한다.

주요 메서드 후보:

```java
List<ChatRoomMemberResponseDto> getMembers(Long loginUserId, Long roomId)
List<ChatRoomMember> getActiveMembers(Long roomId)
```

---

### 6.7 ChatMessageFacade

메시지 전송 use case를 조립한다.

WebSocket 메시지 전송 시 핵심 진입점이다.

주요 메서드 후보:

```java
ChatMessageCreatedEvent sendMessage(Long loginUserId, Long roomId, ChatMessageSendRequestDto request)
```

처리 흐름:

```text
1. 채팅방 멤버 권한 검증
2. 메시지 원문 저장
3. 번역 대상 언어 계산
4. 번역 PENDING 데이터 생성
5. chat.message.created 이벤트 발행
6. chat.translation.pending 이벤트 발행
7. 비동기 번역 요청
8. 생성된 메시지 이벤트 반환
```

---

### 6.8 ChatMessageCommandService

메시지 저장/삭제 등 변경 작업을 담당한다.

주요 메서드 후보:

```java
ChatMessage saveUserMessage(ChatRoom room, User sender, String content)
void deleteMessage(Long messageId, Long loginUserId)
```

---

### 6.9 ChatMessageQueryService

메시지 목록 조회를 담당한다.

주요 메서드 후보:

```java
ChatMessagePageResponseDto getMessages(Long loginUserId, Long roomId, Long cursorMessageId, int limit)
```

조회 기준:

- 채팅방 멤버만 조회 가능
- 최신 메시지 100개 기본 조회
- cursorMessageId 기준 과거 메시지 추가 조회
- 정렬 기준은 `createdAt DESC, id DESC`
- FE 표시 시 시간순 정렬이 필요하면 응답에서 ASC로 재정렬 가능

---

### 6.10 ChatTranslationCommandService

번역 상태 생성/완료/실패 처리를 담당한다.

주요 메서드 후보:

```java
List<ChatMessageTranslation> createPendingTranslations(ChatMessage message, List<String> targetLanguageCodes)
void completeTranslation(Long messageId, String languageCode, String translatedContent)
void failTranslation(Long messageId, String languageCode, String failureReason)
```

---

### 6.11 ChatTranslationDomainService

번역 대상 언어 계산을 담당한다.

주요 메서드 후보:

```java
List<String> resolveTargetLanguageCodes(Long roomId, Long senderUserId)
```

초기 정책:

- 메시지 발신자를 제외한 active member의 `translationLanguageCode`를 수집한다.
- 중복 언어는 제거한다.
- null 또는 원문과 동일한 언어는 제외할 수 있다.
- 1차에서는 원문 언어 자동 감지는 AI Server 또는 후속 처리로 둘 수 있다.

---

### 6.12 ChatTranslationRequestService

AI Server 번역 요청을 담당한다.

주요 메서드 후보:

```java
void requestTranslationAsync(ChatMessage message, List<String> targetLanguageCodes)
```

구현 기준:

- 메시지 저장 트랜잭션과 외부 API 호출을 강하게 묶지 않는다.
- 외부 API 실패가 메시지 저장을 rollback시키지 않도록 한다.
- 초기에는 `@Async` 또는 application event 기반 처리를 검토한다.
- 대규모 Queue는 1차 범위에서 제외한다.

## 7. Controller 설계

### 7.1 ChatRoomController

후보 endpoint:

```text
POST /api/v1/chat/rooms
GET /api/v1/chat/rooms
GET /api/v1/chat/rooms/{roomId}
```

역할:

- 채팅방 생성
- 내 채팅방 목록 조회
- 채팅방 상세 조회

---

### 7.2 ChatRoomMemberController

후보 endpoint:

```text
GET /api/v1/chat/rooms/{roomId}/members
POST /api/v1/chat/rooms/{roomId}/members
DELETE /api/v1/chat/rooms/{roomId}/members/me
PATCH /api/v1/chat/rooms/{roomId}/members/me/language-settings
```

역할:

- 멤버 조회
- 멤버 추가
- 채팅방 나가기
- 채팅방별 언어 설정 변경

---

### 7.3 ChatMessageController

후보 endpoint:

```text
GET /api/v1/chat/rooms/{roomId}/messages
```

역할:

- 메시지 cursor 조회
- 번역 결과 포함 응답

메시지 전송은 REST가 아니라 WebSocket을 우선 사용한다. 단, 테스트 또는 fallback을 위해 REST 전송 API를 별도로 둘지는 후속 검토한다.

---

### 7.4 ChatWebSocketController

후보 destination:

```text
/app/chat/rooms/{roomId}/messages
```

역할:

- 클라이언트 메시지 수신
- 로그인 사용자 식별
- ChatMessageFacade 호출
- 저장된 메시지 이벤트 발행

## 8. WebSocket 발행 설계

### 8.1 Topic

1차 MVP에서는 채팅방 단위 topic을 사용한다.

```text
/topic/chat/rooms/{roomId}
```

### 8.2 Event type

```text
chat.message.created
chat.translation.pending
chat.translation.completed
chat.translation.failed
chat.error
```

### 8.3 Publisher

`ChatEventPublisher`를 두고 WebSocket 발행을 한 곳으로 모은다.

주요 메서드 후보:

```java
void publishMessageCreated(Long roomId, ChatMessageCreatedEvent event)

void publishTranslationFailed(Long roomId, ChatTranslationFailedEvent event)

```

## 9. 트랜잭션 설계

### 9.1 메시지 전송 트랜잭션

메시지 전송 시 DB 저장은 하나의 트랜잭션으로 처리한다.

트랜잭션 포함 후보:

- 채팅방 멤버 검증
- 메시지 원문 저장
- 번역 PENDING 생성

트랜잭션 외부 처리 후보:

- WebSocket 이벤트 발행
- AI Server 번역 요청

권장 흐름:

```text
@Transactional
메시지 저장 + PENDING 생성

트랜잭션 성공 후
WebSocket 원문 이벤트 발행 + 비동기 번역 요청
```

트랜잭션 성공 후 이벤트 처리는 `@TransactionalEventListener(phase = AFTER_COMMIT)` 사용을 검토한다.

## 10. 권한 검증 기준

### 10.1 채팅방 조회

- 로그인 사용자만 가능
- active member만 가능
- 삭제된 채팅방 조회 불가

### 10.2 메시지 조회

- active member만 가능
- 퇴실한 사용자의 과거 메시지 조회 가능 여부는 별도 정책 필요
- 1차에서는 active member 기준으로 제한한다.

### 10.3 메시지 전송

- active member만 가능
- 메시지 content는 빈 값 불가
- 메시지 최대 길이 제한 필요

### 10.4 멤버 초대

- OWNER 또는 ADMIN 권한 필요
- 자기 자신 중복 추가 불가
- 이미 active member인 사용자 중복 추가 불가

## 11. 에러 처리

에러 응답은 기존 TranslaCat 공통 응답 구조를 따른다.

채팅 도메인 주요 에러 후보:

```text
CHAT_ROOM_NOT_FOUND
CHAT_ROOM_ACCESS_DENIED
CHAT_ROOM_MEMBER_NOT_FOUND
CHAT_ROOM_MEMBER_ALREADY_EXISTS
CHAT_MESSAGE_NOT_FOUND
CHAT_MESSAGE_CONTENT_EMPTY
CHAT_MESSAGE_CONTENT_TOO_LONG
CHAT_TRANSLATION_NOT_FOUND
CHAT_TRANSLATION_REQUEST_FAILED
CHAT_WEBSOCKET_SEND_FAILED
```

WebSocket 처리 중 발생한 오류는 가능하면 `chat.error` 이벤트로 전달한다.

## 12. QueryDSL 사용 후보

QueryDSL은 다음 조회에서 사용한다.

### 12.1 내 채팅방 목록 조회

필요 정보:

- roomId
- roomType
- roomName
- lastMessage
- lastMessageAt
- memberCount
- myRole
- myLanguageSetting

### 12.2 메시지 cursor 조회

조건:

```text
roomId = :roomId
message.id < :cursorMessageId   // cursor가 있는 경우
status != DELETED
```

정렬:

```text
createdAt DESC
id DESC
```

limit:

```text
기본 100
최대 100
```

### 12.3 멤버 목록 조회

조건:

```text
roomId = :roomId
active = true
```

## 13. AI Server 연동

1차 BE는 AI Server에 채팅 메시지 번역을 요청한다.

AI Server 호출은 `ChatTranslationClient` 또는 `AiChatTranslationClient`로 분리한다.

후보 메서드:

```java
ChatTranslationClientResponse translate(ChatTranslationClientRequest request)
```

요청 후보:

```json
{
  "sourceLanguageCode": "ko",
```

## 14. 가계부 연동 확장 고려

1차 BE 구현에서는 채팅 도메인과 가계부 도메인을 직접 연결하지 않는다.

금지 방향:

```text
chat_room.account_book_id 직접 추가
```

향후 확장 방향:

```text
chat_room_account_book_link
```

설계 원칙:

- 채팅 권한과 가계부 권한은 분리한다.
- 채팅방 멤버가 가계부 권한을 자동으로 얻지 않는다.
- 실제 가계부 접근은 `account_book_member` 기준으로 검증한다.
- 하나의 채팅방에 여러 가계부를 연동할 수 있도록 한다.
- 하나의 가계부가 여러 채팅방에 공유될 가능성도 고려한다.

## 15. 구현 우선순위

BE 1차 구현 순서는 다음을 권장한다.

```text
1. Entity / Enum / Repository
2. QueryRepository skeleton
3. ChatRoom 생성/목록/상세 API
4. ChatRoomMember 조회/추가/언어 설정 API
5. ChatMessage 조회 API
6. ChatMessage 저장 CommandService
7. WebSocket 설정 및 원문 메시지 송수신
8. 번역 PENDING 생성
9. AI Server 번역 요청 client
10. 번역 완료/실패 저장
11. 번역 WebSocket 이벤트 발행
12. BE 단위/통합 확인
```

## 16. 1차 완료 기준

- 채팅방을 생성할 수 있다.
- 채팅방 목록을 조회할 수 있다.
- 채팅방 상세를 조회할 수 있다.
- 채팅방 멤버를 조회할 수 있다.
- 채팅방별 언어 설정을 변경할 수 있다.
- 최신 메시지 100개를 조회할 수 있다.
- cursor 기준 과거 메시지를 조회할 수 있다.
- WebSocket으로 메시지를 전송할 수 있다.
- 원문 메시지가 DB에 저장된다.
- 원문 메시지가 같은 방 참여자에게 실시간 발행된다.
- 번역 PENDING 데이터가 생성된다.
- AI Server 번역 요청이 수행된다.
- 번역 완료 시 DB에 저장된다.
- 번역 완료 이벤트가 WebSocket으로 발행된다.
- 번역 실패가 메시지 송수신을 막지 않는다.
