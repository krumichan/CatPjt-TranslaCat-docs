# 채팅 기능 상세설계 개요

## 1. 문서 목적

본 문서는 TranslaCat 채팅 기능의 상세설계 기준을 정리한다.

채팅 기능은 사용자가 1:1 또는 그룹 채팅방에서 실시간으로 메시지를 주고받고, 각 사용자의 언어 설정에 따라 메시지 번역을 비동기로 제공하는 것을 목표로 한다.

1차 개발에서는 채팅 기능의 핵심 흐름을 MVP 범위로 구현하고, 2차 이후 오픈 채팅, AI 채팅 멤버, 읽음/미읽음, 온라인 상태, 알림 기능을 확장한다.

또한 향후 그룹 채팅방 또는 오픈 채팅방에 가계부를 연동하여 정산, 회비, 공동 지출 관리 등을 지원할 수 있도록 확장 가능성을 고려한다.

---

## 2. MVP 정의

MVP는 Minimum Viable Product의 약자로, 서비스의 핵심 가치를 검증할 수 있는 최소 기능 버전을 의미한다.

TranslaCat 채팅 기능의 1차 MVP는 완성형 메신저를 만드는 것이 아니라, 다음 흐름이 실제로 동작하는지 검증하는 것을 목표로 한다.

```text
사용자가 채팅방을 생성한다.
사용자가 메시지를 전송한다.
상대방이 메시지를 실시간으로 수신한다.
원문 메시지는 즉시 표시된다.
번역 메시지는 비동기로 생성되어 나중에 표시된다.
```

즉, 1차 MVP의 핵심은 “실시간 원문 채팅 + 비동기 번역 표시”이다.

## 8. Backend 설계 방향

Backend는 단순한 Controller-Service-Repository 구조만 사용하지 않고, 기능 복잡도에 따라 Facade, CommandService, QueryService, DomainService, QueryRepository를 분리한다.

### 8.1 기본 계층

```text
Controller
  ↓
Facade
  ↓
CommandService / QueryService / DomainService
  ↓
Repository / QueryRepository / External Client
```

### 8.2 계층별 역할

#### Controller

- HTTP 요청 또는 WebSocket 메시지를 수신한다.
- request DTO를 검증한다.
- 인증 사용자 정보를 service layer에 전달한다.
- 비즈니스 흐름을 직접 처리하지 않는다.

#### Facade

- 하나의 사용자 흐름을 조립한다.
- 여러 service를 호출한다.
- 트랜잭션 경계 후보가 된다.
- 복합 use case를 표현한다.

예시:

- 채팅방 생성
- 메시지 전송
- 메시지 저장 후 WebSocket 이벤트 발행
- 메시지 저장 후 번역 요청 시작

#### CommandService

- DB 상태를 변경한다.
- 생성, 수정, 삭제 처리를 담당한다.
- Entity 생성 및 상태 변경을 담당한다.

#### QueryService

- 조회 전용 로직을 담당한다.
- 목록, 상세, cursor 조회 등 복잡한 조회 조건을 처리한다.
- QueryDSL 기반 조회를 사용할 수 있다.

#### DomainService

- 도메인 규칙을 처리한다.
- 특정 Entity 하나에 넣기 어려운 정책을 담당한다.

예시:

- 채팅방 접근 권한 검증
- 메시지 전송 가능 여부 검증
- 번역 대상 언어 계산
- 채팅방 멤버 역할 검증

#### Repository

- JPA 기반 기본 persistence 처리를 담당한다.

#### QueryRepository

- QueryDSL 기반 복잡한 조회를 담당한다.
- 목록 조회, cursor 조회, 검색 조건 조합 등에 사용한다.

#### External Client

- AI Server 또는 외부 API 호출을 담당한다.

#### Event Publisher

- WebSocket 이벤트 발행을 담당한다.
- 이벤트 payload 생성과 발행 책임을 분리한다.

## 11. 메시지 송수신 기본 흐름

### 11.1 사용자 메시지 전송

```text
Client
→ WebSocket send
→ Backend WebSocket Controller
→ ChatMessageFacade
→ 채팅방 멤버 권한 검증
→ 메시지 원문 저장
→ 번역 대상 언어 계산
→ 번역 PENDING 데이터 생성
→ chat.message.created 발행
→ chat.translation.pending 발행
→ 비동기 번역 요청
```

### 11.2 메시지 원문 수신

```text
Backend
→ /topic/chat/rooms/{roomId}
→ chat.message.created
→ Client 메시지 목록에 원문 표시
```

### 11.3 번역 완료

```text
Backend Async Translation Worker
→ AI Server 호출
→ 번역 결과 수신
→ chat_message_translation 갱신
→ chat.translation.completed 발행
→ Client 기존 메시지의 번역 영역 갱신
```

### 11.4 번역 실패

```text
Backend Async Translation Worker
→ AI Server 호출 실패 또는 번역 실패
→ chat_message_translation FAILED 처리
→ chat.translation.failed 발행
→ Client 번역 실패 상태 표시
```

## 14. 메시지 조회 정책

메시지 조회는 cursor 기반 keyset pagination을 사용한다.

기본 정책:

```text
초기 진입: 최신 메시지 100개 조회
위로 스크롤: 이전 메시지 100개 추가 조회
정렬 기준: createdAt DESC, id DESC
FE 표시: 시간순 ASC로 변환 가능
```

API 후보:

```text
GET /api/v1/chat/rooms/{roomId}/messages?cursorMessageId={cursorMessageId}&limit=100
```

주의사항:

- 중복 메시지를 제거한다.
- cursor 기준 메시지보다 오래된 메시지를 조회한다.
- 삭제된 메시지 표시 정책을 별도로 정한다.
- 채팅방 멤버가 아닌 사용자는 조회할 수 없다.

## 15. 번역 처리 정책

번역 처리는 메시지 송수신과 분리하여 비동기로 처리한다.

기본 정책:

- 원문 메시지는 즉시 저장한다.
- 원문 메시지는 즉시 WebSocket으로 발행한다.
- 번역 대상 언어를 계산한다.
- 번역 대기 데이터를 `PENDING` 상태로 생성한다.
- AI Server에 번역을 요청한다.
- 번역 성공 시 `COMPLETED`로 변경한다.
- 번역 실패 시 `FAILED`로 변경한다.
- 번역 실패가 원문 메시지 송수신을 막지 않는다.

번역 대상 언어 계산 기준 후보:

```text
같은 채팅방 멤버들의 translationLanguageCode
중복 제거
원문 언어와 동일한 언어 제외
비활성 또는 퇴실 멤버 제외
```

## 17. 향후 가계부 연동 고려

채팅방과 가계부는 서로 다른 도메인으로 관리한다.

1차 개발에서는 채팅방과 가계부를 직접 연결하지 않는다.

향후 그룹 채팅방 또는 오픈 채팅방에 하나 이상의 가계부를 연동할 수 있도록 별도 연결 테이블 도입을 고려한다.

### 17.1 연결 테이블 후보

```text
chat_room_account_book_link
```

### 17.2 관계

```text
chat_room 1 ─ N chat_room_account_book_link N ─ 1 account_book
```

### 17.3 주요 컬럼 후보

```text
id
chat_room_id
account_book_id
linked_by_user_id
link_status
display_order
created_at
deleted_at
```

### 17.4 설계 원칙

- 채팅방과 가계부는 강하게 결합하지 않는다.
- `chat_room`에 `accountBookId`를 직접 두지 않는다.
- 하나의 채팅방에 여러 가계부를 연동할 수 있도록 한다.
- 하나의 가계부가 여러 채팅방에 공유될 가능성도 고려한다.
- 채팅방에 가계부를 연동해도 채팅방 멤버가 자동으로 가계부 권한을 얻지는 않는다.
- 가계부 접근 권한은 기존 `account_book_member` 기준으로 판단한다.
- 채팅방 권한과 가계부 권한은 분리한다.
- 필요 시 후속 개발에서 채팅방 멤버를 가계부 멤버로 초대하는 기능을 제공한다.

## 18. 1차 개발 진행 순서

권장 개발 순서는 다음과 같다.

```text

3. BE 채팅방 API 구현
4. BE 채팅방 멤버/언어 설정 API 구현

6. FE 채팅 feature 구조/type/api client 구현
7. FE 채팅방 목록/생성/상세 기본 UI 구현
8. BE WebSocket 원문 메시지 송수신 구현
9. FE WebSocket 연결/메시지 송수신 구현
10. AI 채팅 번역 API/prompt 구현
11. BE 비동기 번역 처리 구현
12. FE 번역 상태/번역문 표시 구현
13. FE 과거 메시지 추가 로딩 구현
14. 1차 QA 결합 확인
```

---

## 19. 설계 시 주의사항

- 메시지 원문 저장과 번역 처리는 분리한다.
- 원문 메시지는 즉시 저장 및 전송한다.
- 번역 처리는 비동기로 수행한다.
- 번역 실패가 메시지 송수신을 막지 않도록 한다.
- 채팅방 멤버가 아닌 사용자는 메시지를 조회하거나 전송할 수 없다.
- cursor 기반 메시지 조회를 사용한다.
- WebSocket 이벤트명은 FE/BE 간 사전에 고정한다.
- 향후 Redis 도입 가능성을 고려하되, 1차에서는 MySQL 기반으로 구현한다.
- 향후 가계부 연동을 위해 채팅 도메인에 `accountBookId`를 직접 넣지 않는다.
- 2차 기능이 1차 MVP 구현을 방해하지 않도록 범위를 분리한다.
