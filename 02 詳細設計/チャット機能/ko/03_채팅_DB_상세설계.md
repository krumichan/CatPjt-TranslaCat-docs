# 채팅 DB 상세설계

## 1. 문서 목적

본 문서는 TranslaCat 채팅 기능 1차 MVP 구현을 위한 DB 상세설계를 정의한다.

1차 MVP의 DB 설계 대상은 다음 4개 테이블이다.

```text
chat_room
chat_room_member
chat_message
chat_message_translation
```

2차 이후 확장 후보인 오픈 채팅, AI 채팅 멤버, 읽음/미읽음, 온라인 상태, 알림, Redis, 가계부 연동은 별도 테이블로 확장한다.

---

## 2. 공통 설계 원칙

- PK는 `BIGINT` auto increment를 기본으로 한다.
- 생성/수정 시각은 기존 프로젝트 공통 BaseEntity 정책을 따른다.
- 삭제는 가능하면 soft delete를 사용한다.
- enum은 문자열 저장을 기본으로 한다.
- 대량 조회가 예상되는 테이블은 cursor 조회 기준 index를 둔다.
- 채팅 메시지는 수정/삭제보다 조회/추가가 많으므로 append 중심으로 설계한다.

---

## 3. chat_room

채팅방 기본 정보를 관리한다.

| 컬럼 | 타입 | NULL | 설명 |
|---|---:|---:|---|
| id | BIGINT | N | PK |
| room_type | VARCHAR(30) | N | DIRECT, GROUP, OPEN |
| name | VARCHAR(100) | Y | 채팅방 이름 |
| description | VARCHAR(500) | Y | 설명 |
| owner_id | BIGINT | Y | 생성자/방장 user id |
| active | BOOLEAN | N | 사용 여부 |
| created_at | DATETIME | N | 생성일시 |
| updated_at | DATETIME | Y | 수정일시 |
| deleted_at | DATETIME | Y | 삭제일시 |

### 3.1 room_type

1차 MVP에서 사용하는 값:

```text
DIRECT
GROUP
```

2차 이후 확장 값:

```text
OPEN
```

### 3.2 index 후보

```sql
CREATE INDEX idx_chat_room_owner_id ON chat_room(owner_id);
CREATE INDEX idx_chat_room_room_type ON chat_room(room_type);
```

---

## 4. chat_room_member

채팅방 참여자와 언어 설정을 관리한다.

| 컬럼 | 타입 | NULL | 설명 |
|---|---:|---:|---|
| id | BIGINT | N | PK |
| chat_room_id | BIGINT | N | 채팅방 ID |
| user_id | BIGINT | N | 사용자 ID |
| role | VARCHAR(30) | N | OWNER, ADMIN, MEMBER |
| original_language_code | VARCHAR(10) | Y | 원문 언어 코드 |
| translation_language_code | VARCHAR(10) | Y | 번역 표시 언어 코드 |
| show_original | BOOLEAN | N | 원문 표시 여부 |
| show_translation | BOOLEAN | N | 번역 표시 여부 |
| active | BOOLEAN | N | 참여 중 여부 |
| joined_at | DATETIME | N | 참여일시 |
| left_at | DATETIME | Y | 퇴실일시 |
| created_at | DATETIME | N | 생성일시 |
| updated_at | DATETIME | Y | 수정일시 |
| deleted_at | DATETIME | Y | 삭제일시 |

### 4.1 unique 제약

```sql
CREATE UNIQUE INDEX uk_chat_room_member_room_user
ON chat_room_member(chat_room_id, user_id);
```

soft delete 후 재참여를 허용할 경우 unique 제약과 active/deleted_at 정책을 별도 검토한다.
초기 구현에서는 기존 row를 restore하는 방식을 우선 검토한다.

### 4.2 index 후보

```sql
CREATE INDEX idx_chat_room_member_user_active
ON chat_room_member(user_id, active);

CREATE INDEX idx_chat_room_member_room_active
ON chat_room_member(chat_room_id, active);
```

## 5. chat_message

채팅 메시지 원문을 저장한다.

| 컬럼 | 타입 | NULL | 설명 |
|---|---:|---:|---|
| id | BIGINT | N | PK |
| chat_room_id | BIGINT | N | 채팅방 ID |
| sender_user_id | BIGINT | Y | 발신 사용자 ID |
| sender_type | VARCHAR(30) | N | USER, AI |
| message_type | VARCHAR(30) | N | TEXT, SYSTEM |
| content | TEXT | N | 원문 내용 |
| status | VARCHAR(30) | N | SENT, DELETED |
| created_at | DATETIME | N | 생성일시 |
| updated_at | DATETIME | Y | 수정일시 |
| deleted_at | DATETIME | Y | 삭제일시 |

### 5.1 sender_type

1차 MVP:

```text
USER
```

2차 이후:

```text
AI
```

AI 메시지 확장을 고려하여 `sender_user_id`는 nullable로 둔다. 단, 1차 USER 메시지에서는 반드시 값이 있어야 한다.

### 5.2 index 후보

메시지 cursor 조회를 위해 다음 index를 둔다.

```sql
CREATE INDEX idx_chat_message_room_created_id
ON chat_message(chat_room_id, created_at, id);
```

cursorMessageId 기준 조회를 주로 사용할 경우 다음 index도 검토한다.

```sql
CREATE INDEX idx_chat_message_room_id_id
ON chat_message(chat_room_id, id);
```

## 6. chat_message_translation

| 컬럼 | 타입 | NULL | 설명 |
|---|---:|---:|---|

| updated_at | DATETIME | Y | 수정일시 |
| completed_at | DATETIME | Y | 완료일시 |

### 6.1 unique 제약

```sql
CREATE UNIQUE INDEX uk_chat_message_translation_message_language
ON chat_message_translation(chat_message_id, language_code);
```

### 6.2 index 후보

```sql
CREATE INDEX idx_chat_translation_message_id
ON chat_message_translation(chat_message_id);

CREATE INDEX idx_chat_translation_status
ON chat_message_translation(status);
```

## 8. 가계부 연동 확장 테이블 후보

채팅방과 가계부는 직접 결합하지 않는다.

금지:

```text
chat_room.account_book_id
```

권장:

```text
chat_room_account_book_link
```

후보 컬럼:

| 컬럼 | 설명 |
|---|---|
| id | PK |
| chat_room_id | 채팅방 ID |
| account_book_id | 가계부 ID |
| linked_by_user_id | 연동한 사용자 ID |
| link_status | ACTIVE, REMOVED |
| display_order | 표시 순서 |
| created_at | 생성일시 |
| deleted_at | 삭제일시 |

설계 원칙:

- 채팅방 멤버 권한과 가계부 멤버 권한은 분리한다.
- 가계부 접근은 `account_book_member` 기준으로 검증한다.
- 하나의 채팅방에 여러 가계부를 연결할 수 있다.
- 하나의 가계부가 여러 채팅방에 연결될 가능성도 고려한다.

## 9. 데이터 조회 정책

### 9.1 메시지 첫 조회

```text
최신 메시지 100개
```

정렬:

```text
DB 조회: created_at DESC, id DESC
FE 표시: created_at ASC, id ASC
```

### 9.2 과거 메시지 조회

cursor 기준:

```text
cursorMessageId
```

조건 후보:

```sql
WHERE chat_room_id = :roomId
  AND id < :cursorMessageId
ORDER BY id DESC
LIMIT :limit
```

createdAt/id 복합 cursor를 쓸 수도 있으나, 1차에서는 message id cursor를 우선 검토한다.

---

## 10. 삭제 정책

### 10.1 채팅방 삭제

1차에서는 실제 삭제 기능을 구현하지 않아도 된다.
추후 삭제 시 soft delete를 사용한다.

### 10.2 멤버 퇴실

멤버 퇴실은 row 삭제보다 active false + leftAt 기록을 우선 고려한다.

### 10.3 메시지 삭제

메시지는 실제 삭제하지 않고 status를 `DELETED`로 변경하는 방식을 우선 고려한다.
