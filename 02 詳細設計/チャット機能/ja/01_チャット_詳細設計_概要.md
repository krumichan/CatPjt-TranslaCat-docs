# チャット機能詳細設計概要

## 1. 文書目的

本書は、TranslaCatチャット機能の詳細設計基準を整理するための文書である。

チャット機能は、ユーザーが1:1またはグループチャットルームでリアルタイムにメッセージを送受信し、各ユーザーの言語設定に応じてメッセージ翻訳を非同期で提供することを目的とする。

1次開発ではチャット機能の中核フローをMVP範囲として実装し、2次以降でオープンチャット、AIチャットメンバー、既読/未読、オンライン状態、通知機能を拡張する。

また、将来的にグループチャットルームまたはオープンチャットルームへ家計簿を連携し、精算、会費、共同支出管理などを支援できるよう、拡張可能性を考慮する。

---

## 2. MVP定義

MVPは Minimum Viable Product の略で、サービスの中核価値を検証できる最小機能バージョンを意味する。

TranslaCatチャット機能の1次MVPは、完成形のメッセンジャーを作ることではなく、次のフローが実際に動作することを検証することを目的とする。

```text
ユーザーがチャットルームを作成する。
ユーザーがメッセージを送信する。
相手がメッセージをリアルタイムに受信する。
原文メッセージは即時表示される。
翻訳メッセージは非同期で生成され、後から表示される。
```

つまり、1次MVPの中核は「リアルタイム原文チャット + 非同期翻訳表示」である。

## 3. 1次開発範囲

1次開発範囲は以下の通りである。

- 1:1チャット
- グループチャット
- チャットルーム作成
- チャットルーム一覧取得
- チャットルーム詳細取得
- チャットルームメンバー取得
- チャットルームメンバー別言語設定
- メッセージ保存
- メッセージ取得
- WebSocketベースのリアルタイムメッセージ送受信
- 原文メッセージ即時表示
- メッセージ非同期翻訳
- 翻訳状態表示
- 翻訳完了イベント受信
- 最新メッセージ100件取得
- 上スクロール時の過去メッセージ100件追加取得

---

## 4. 1次開発対象外

以下の機能は1次開発範囲から除外する。

- オープンチャット
- AIチャットメンバー
- AI応答生成
- 既読/未読
- オンライン状態
- チャット通知
- Redisベースのscale-out
- 家計簿連携
- 精算機能
- ブラウザ通知
- モバイルPush通知

ただし、家計簿連携は将来的な拡張可能性が高いため、1次設計段階でチャットドメインと家計簿ドメインが強く結合しないよう注意する。

## 8. Backend設計方針

Backendは単純なController-Service-Repository構成だけではなく、機能の複雑度に応じてFacade、CommandService、QueryService、DomainService、QueryRepositoryを分離する。

### 8.1 基本レイヤー

```text
Controller
  ↓
Facade
  ↓
CommandService / QueryService / DomainService
  ↓
Repository / QueryRepository / External Client
```

### 8.2 レイヤー別役割

#### Controller

- HTTPリクエストまたはWebSocketメッセージを受信する。
- request DTOを検証する。
- 認証ユーザー情報をservice layerへ渡す。
- ビジネスフローを直接処理しない。

#### Facade

- 1つのユーザーフローを組み立てる。
- 複数serviceを呼び出す。
- トランザクション境界候補となる。
- 複合use caseを表現する。

例:

- チャットルーム作成
- メッセージ送信
- メッセージ保存後のWebSocketイベント配信
- メッセージ保存後の翻訳依頼開始

#### CommandService

- DB状態を変更する。
- 作成、更新、削除処理を担当する。
- Entity生成および状態変更を担当する。

#### QueryService

- 参照専用ロジックを担当する。
- 一覧、詳細、cursor取得など複雑な検索条件を処理する。
- QueryDSLベースの参照を使用できる。

#### DomainService

- ドメインルールを処理する。
- 特定Entity単体に入れにくいポリシーを担当する。

例:

- チャットルームアクセス権限検証
- メッセージ送信可否検証
- 翻訳対象言語計算
- チャットルームメンバー役割検証

#### Repository

- JPAベースの基本persistence処理を担当する。

#### QueryRepository

- QueryDSLベースの複雑な参照を担当する。
- 一覧取得、cursor取得、検索条件組み合わせなどに使用する。

#### External Client

- AI Serverまたは外部API呼び出しを担当する。

#### Event Publisher

- WebSocketイベント配信を担当する。
- イベントpayload生成と配信責務を分離する。

## 11. メッセージ送受信基本フロー

### 11.1 ユーザーメッセージ送信

```text
Client
→ WebSocket send
→ Backend WebSocket Controller
→ ChatMessageFacade
→ チャットルームメンバー権限検証
→ 原文メッセージ保存
→ 翻訳対象言語計算
→ 翻訳PENDINGデータ作成
→ chat.message.created配信
→ chat.translation.pending配信
→ 非同期翻訳依頼
```

### 11.2 原文メッセージ受信

```text
Backend
→ /topic/chat/rooms/{roomId}
→ chat.message.created
→ Clientメッセージ一覧に原文表示
```

### 11.3 翻訳完了

```text
Backend Async Translation Worker
→ AI Server呼び出し
→ 翻訳結果受信
→ chat_message_translation更新
→ chat.translation.completed配信
→ Client既存メッセージの翻訳領域更新
```

### 11.4 翻訳失敗

```text
Backend Async Translation Worker
→ AI Server呼び出し失敗または翻訳失敗
→ chat_message_translation FAILED処理
→ chat.translation.failed配信
→ Client翻訳失敗状態表示
```

## 14. メッセージ取得ポリシー

メッセージ取得はcursorベースのkeyset paginationを使用する。

基本方針:

```text
初期入室: 最新メッセージ100件取得
上スクロール: 過去メッセージ100件追加取得
ソート基準: createdAt DESC, id DESC
FE表示: 時系列ASCへ変換可能
```

API候補:

```text
GET /api/v1/chat/rooms/{roomId}/messages?cursorMessageId={cursorMessageId}&limit=100
```

注意事項:

- 重複メッセージを除去する。
- cursor基準メッセージより古いメッセージを取得する。
- 削除済みメッセージ表示ポリシーを別途定義する。
- チャットルームメンバーではないユーザーは取得できない。

---

## 15. 翻訳処理ポリシー

翻訳処理はメッセージ送受信と分離し、非同期で処理する。

基本方針:

- 原文メッセージは即時保存する。
- 原文メッセージは即時WebSocketで配信する。
- 翻訳対象言語を計算する。
- 翻訳待ちデータを `PENDING` 状態で作成する。
- AI Serverへ翻訳を依頼する。
- 翻訳成功時 `COMPLETED` へ変更する。
- 翻訳失敗時 `FAILED` へ変更する。
- 翻訳失敗が原文メッセージ送受信を妨げない。

翻訳対象言語計算基準候補:

```text
同一チャットルームメンバーのtranslationLanguageCode
重複除去
原文言語と同一言語は除外
非activeまたは退出済みメンバーは除外
```

## 17. 将来の家計簿連携考慮

チャットルームと家計簿は別ドメインとして管理する。

1次開発ではチャットルームと家計簿を直接連携しない。

将来的にグループチャットルームまたはオープンチャットルームに1つ以上の家計簿を連携できるよう、別途連携テーブル導入を検討する。

### 17.1 連携テーブル候補

```text
chat_room_account_book_link
```

### 17.2 関係

```text
chat_room 1 ─ N chat_room_account_book_link N ─ 1 account_book
```

### 17.3 主なカラム候補

```text
id
chat_room_id
account_book_id
linked_by_user_id
link_status
display_order
created_at
deleted_at
```

### 17.4 設計原則

- チャットルームと家計簿を強く結合しない。
- `chat_room` に `accountBookId` を直接持たせない。
- 1つのチャットルームに複数家計簿を連携できるようにする。
- 1つの家計簿が複数チャットルームに共有される可能性も考慮する。
- チャットルームに家計簿を連携しても、チャットルームメンバーが自動で家計簿権限を得るわけではない。
- 家計簿アクセス権限は既存の `account_book_member` を基準に判断する。
- チャットルーム権限と家計簿権限は分離する。
- 必要に応じて後続開発でチャットルームメンバーを家計簿メンバーへ招待する機能を提供する。

## 18. 1次開発進行順序

推奨開発順序は以下の通りである。

```text
1. 詳細設計ドキュメント確定
2. BE Entity/Repository実装
3. BEチャットルームAPI実装
4. BEチャットルームメンバー/言語設定API実装
5. BEメッセージ取得/保存service実装
6. FEチャットfeature構成/type/api client実装
7. FEチャットルーム一覧/作成/詳細基本UI実装
8. BE WebSocket原文メッセージ送受信実装
9. FE WebSocket接続/メッセージ送受信実装
10. AIチャット翻訳API/prompt実装
11. BE非同期翻訳処理実装
12. FE翻訳状態/翻訳文表示実装
13. FE過去メッセージ追加取得実装
14. 1次QA結合確認
```

---

## 19. 設計時の注意事項

- メッセージ原文保存と翻訳処理は分離する。
- 原文メッセージは即時保存および配信する。
- 翻訳処理は非同期で実行する。
- 翻訳失敗がメッセージ送受信を妨げないようにする。
- チャットルームメンバーではないユーザーはメッセージを取得または送信できない。
- cursorベースのメッセージ取得を使用する。
- WebSocketイベント名はFE/BE間で事前に固定する。
- 将来のRedis導入可能性を考慮するが、1次ではMySQLベースで実装する。
- 将来の家計簿連携のため、チャットドメインに `accountBookId` を直接入れない。
- 2次機能が1次MVP実装を妨げないよう範囲を分離する。
