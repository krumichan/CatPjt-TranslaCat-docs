# チャット 家計簿連携 拡張設計

## 1. 文書目的

本書は、TranslaCatチャット機能と家計簿機能の将来連携可能性を整理する。

家計簿連携は1次チャットMVP範囲ではなく、2次または3次以降の拡張候補として管理する。
ただし1次チャットドメイン設計で家計簿連携を妨げる強い結合を作らないため、本書に拡張方針を残す。

---

## 2. 連携目的

チャットルームと家計簿を連携すると、以下のユースケースを支援できる。

- 友人グループチャットの旅行精算
- サークル/コミュニティの会費管理
- 集まり別共同支出記録
- 飲み会費/宿泊費/交通費精算
- レシート分析結果をチャットルームに共有
- チャットメッセージから支出登録候補を生成

---

## 3. 設計原則

最も重要な原則は以下である。

```text
チャットルームと家計簿を直接結合しない。
```

禁止方向:

```text
chat_room.account_book_id
```

推奨方向:

```text
chat_room_account_book_link
```

---

## 4. 関係設計

```text
chat_room 1 ─ N chat_room_account_book_link N ─ 1 account_book
```

この構造により、以下を満たすことができる。

- 1つのチャットルームに複数家計簿を連携
- 1つの家計簿を複数チャットルームへ共有
- 連携状態soft delete
- 表示順管理
- 連携したユーザー記録

---

## 5. 連携テーブル候補

### chat_room_account_book_link

| カラム | 説明 |
|---|---|
| id | PK |
| chat_room_id | チャットルームID |
| account_book_id | 家計簿ID |
| linked_by_user_id | 連携したユーザーID |
| link_status | ACTIVE, REMOVED |
| display_order | チャットルーム内表示順 |
| created_at | 作成日時 |
| deleted_at | 削除日時 |

unique候補:

```sql
CREATE UNIQUE INDEX uk_chat_room_account_book_link
ON chat_room_account_book_link(chat_room_id, account_book_id);
```

---

## 6. 権限原則

チャット権限と家計簿権限は分離する。

```text
チャットルームメンバーだからといって家計簿メンバーになるわけではない。
家計簿メンバーだからといってチャットルームメンバーになるわけでもない。
```

家計簿アクセス時は必ず既存家計簿権限を確認する。

```text
account_book_member
```

---

## 7. 後続機能候補

### 7.1 家計簿連携

チャットルーム設定で家計簿を連携する。

### 7.2 家計簿招待連携

家計簿をチャットルームに連携する際、チャットルームメンバーを家計簿メンバーとして招待する。

### 7.3 精算結果共有

家計簿の月別/項目別精算結果をチャットメッセージとして共有する。

### 7.4 レシート分析共有

AIレシート分析結果を特定チャットルームへ共有する。

### 7.5 チャットベース取引候補生成

例:

```text
A: 今日の飲み会費12000円は私が払ったよ
B: じゃあ4人で割ろう
```

AIがこのメッセージをもとに取引登録候補を提案する。

---

## 8. 開発段階案

```text
1次: チャットMVP実装、家計簿連携未実装
2次: オープンチャット/AI/既読/通知/Redis
3次: チャットルーム・家計簿連携テーブル導入
4次: 精算/AI支出候補/レシート共有拡張
```

---

## 9. 1次設計で守ること

- `chat_room` に `account_book_id` を入れない。
- チャットルームメンバー権限と家計簿メンバー権限を混ぜない。
- チャットメッセージドメインが家計簿取引ドメインに直接依存しないようにする。
- 将来システムメッセージとして家計簿イベントを表示できるよう `ChatMessageType.SYSTEM` を開けておく。
