# チャット BE 詳細設計

## 1. 文書目的

本書は、TranslaCatチャット機能の1次MVP実装に向けたBackend詳細設計を定義する。

1次MVPでは、以下の流れを優先して実装する。

- チャットルーム作成
- チャットルーム一覧/詳細取得
- チャットルームメンバーおよび言語設定管理
- メッセージ保存/取得
- WebSocketによるリアルタイムメッセージ送受信
- 原文メッセージ即時配信
- メッセージ翻訳非同期処理
- 翻訳完了/失敗イベント配信

2次以降の拡張機能であるオープンチャット、AIチャットメンバー、既読/未読、オンライン状態、通知、Redis、家計簿連携は1次実装範囲外とする。ただし、将来拡張を妨げないようドメイン結合度を低く保つ。

---

## 2. 設計原則

### 2.1 1次MVP優先原則

1次開発では、チャット機能の中核フローを検証することに集中する。

```text
ユーザーメッセージ送信
→ 原文メッセージDB保存
→ WebSocketで原文即時配信
→ 翻訳待ち状態作成
→ AI Serverへ非同期翻訳リクエスト
→ 翻訳結果保存
→ WebSocketで翻訳完了イベント配信
```

以下は1次では実装しない。

- オープンチャット検索/参加
- AIチャットメンバー応答
- 既読/未読
- オンライン状態
- チャット通知
- Redisベースscale-out
- チャットルーム・家計簿連携

---

### 2.2 レイヤー分離原則

チャットドメインは単純CRUDより処理フローが長くなるため、Controller-Service-Repository構造だけに限定しない。
機能の複雑度に応じて以下のレイヤーを使用する。

```text
Controller / WebSocketController
  ↓
Facade
  ↓
CommandService / QueryService / DomainService
  ↓
Repository / QueryRepository / External Client
  ↓
Event Publisher
```

| レイヤー | 役割 |
|---|---|
| Controller | HTTPリクエスト受信、DTO検証、認証ユーザー伝達 |
| WebSocketController | STOMPメッセージ受信、メッセージ送信要求処理 |
| Facade | 1つのuse caseを組み立て、複数serviceを呼び出す |
| CommandService | 作成/修正/削除などDB変更処理 |
| QueryService | 取得専用処理、QueryDSLベース取得呼び出し |
| DomainService | 権限、ポリシー、翻訳対象計算などドメイン規則処理 |
| Repository | JPA基本persistence処理 |
| QueryRepository | QueryDSLベースの複雑な取得処理 |
| External Client | AI Serverなど外部API呼び出し |
| Event Publisher | WebSocketイベント配信 |

## 3. パッケージ構成候補

```text
jp.co.translacat.domain.chat
├─ room
├─ member
├─ message
├─ translation
├─ websocket
└─ client
   └─ ai
```

最低限、以下の責務分離は維持する。

- チャットルーム関連: `room`
- メンバー/権限/言語設定: `member`
- 原文メッセージ: `message`
- メッセージ翻訳: `translation`
- WebSocketイベント: `websocket`
- AI Server呼び出し: `client.ai`

## 5. 主要Service設計

### 5.1 ChatRoomFacade

チャットルーム作成などのuse caseを組み立てる。

```java
ChatRoomResponseDto createRoom(Long loginUserId, ChatRoomCreateRequestDto request)
ChatRoomDetailResponseDto getRoomDetail(Long loginUserId, Long roomId)
```

### 5.2 ChatRoomAccessService

チャットルームアクセス権限を検証する。

```java
ChatRoomMember validateRoomMember(Long roomId, Long userId)
void validateCanSendMessage(Long roomId, Long userId)
void validateCanManageRoom(Long roomId, Long userId)
```

### 5.3 ChatMessageFacade

WebSocketメッセージ送信時の中核use caseを担当する。

```java
ChatMessageCreatedEvent sendMessage(Long loginUserId, Long roomId, ChatMessageSendRequestDto request)
```

処理フロー:

```text
1. チャットルームメンバー権限検証
2. 原文メッセージ保存
3. 翻訳対象言語計算
4. 翻訳PENDINGデータ作成
5. chat.message.createdイベント配信
6. chat.translation.pendingイベント配信
7. 非同期翻訳リクエスト
```

### 5.4 ChatMessageQueryService

メッセージ一覧取得を担当する。

```java
ChatMessagePageResponseDto getMessages(Long loginUserId, Long roomId, Long cursorMessageId, int limit)
```

取得基準:

- active memberのみ取得可能
- 最新100件を基本取得
- cursorMessageId基準で過去メッセージ追加取得
- `createdAt DESC, id DESC` を基本ソートとする

### 5.5 ChatTranslationCommandService

翻訳状態の作成/完了/失敗処理を担当する。

```java
List<ChatMessageTranslation> createPendingTranslations(ChatMessage message, List<String> targetLanguageCodes)
void completeTranslation(Long messageId, String languageCode, String translatedContent)
void failTranslation(Long messageId, String languageCode, String failureReason)
```

## 8. トランザクション設計

メッセージ送信時、DB保存は1つのtransactionで処理する。

transaction対象:

- メンバー権限検証
- 原文メッセージ保存
- 翻訳PENDING作成

transaction外候補:

- WebSocketイベント配信
- AI Server翻訳リクエスト

推奨:

```text
@Transactional
メッセージ保存 + PENDING作成

commit成功後
WebSocketイベント配信 + 非同期翻訳リクエスト
```

`@TransactionalEventListener(phase = AFTER_COMMIT)` の使用を検討する。

## 10. 家計簿連携拡張考慮

1次BE実装ではチャットドメインと家計簿ドメインを直接連携しない。

禁止:

```text
chat_room.account_book_id
```

将来拡張:

```text
chat_room_account_book_link
```

原則:

- チャット権限と家計簿権限は分離する。
- 家計簿アクセスは `account_book_member` を基準に検証する。
- 1つのチャットルームに複数家計簿を連携可能にする。
