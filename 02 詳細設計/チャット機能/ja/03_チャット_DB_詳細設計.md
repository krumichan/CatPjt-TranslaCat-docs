# チャット DB 詳細設計

## 1. 文書目的

本書は、TranslaCatチャット機能1次MVP実装に向けたDB詳細設計を定義する。

1次MVPのDB設計対象は以下の4テーブルである。

```text
chat_room
chat_room_member
chat_message
chat_message_translation
```

2次以降のオープンチャット、AIチャットメンバー、既読/未読、オンライン状態、通知、Redis、家計簿連携は別テーブルで拡張する。

---

## 2. 共通設計原則

- PKは `BIGINT` auto incrementを基本とする。
- 作成/更新日時は既存プロジェクト共通BaseEntity方針に従う。
- 削除は可能な限りsoft deleteを使用する。
- enumは文字列保存を基本とする。
- 大量取得が想定されるテーブルにはcursor取得用indexを設ける。
- チャットメッセージはappend中心で設計する。

## 3. chat_room

| カラム | 型 | NULL | 説明 |
|---|---:|---:|---|
| id | BIGINT | N | PK |
| room_type | VARCHAR(30) | N | DIRECT, GROUP, OPEN |
| name | VARCHAR(100) | Y | チャットルーム名 |
| description | VARCHAR(500) | Y | 説明 |
| owner_id | BIGINT | Y | 作成者/オーナーuser id |
| active | BOOLEAN | N | 使用可否 |
| created_at | DATETIME | N | 作成日時 |
| updated_at | DATETIME | Y | 更新日時 |
| deleted_at | DATETIME | Y | 削除日時 |

1次MVP:

```text
DIRECT
GROUP
```

2次以降:

```text
OPEN
```

index候補:

```sql
CREATE INDEX idx_chat_room_owner_id ON chat_room(owner_id);
CREATE INDEX idx_chat_room_room_type ON chat_room(room_type);
```

## 4. chat_room_member

| カラム | 型 | NULL | 説明 |
|---|---:|---:|---|
| id | BIGINT | N | PK |
| chat_room_id | BIGINT | N | チャットルームID |
| user_id | BIGINT | N | ユーザーID |
| role | VARCHAR(30) | N | OWNER, ADMIN, MEMBER |
| original_language_code | VARCHAR(10) | Y | 原文言語コード |
| translation_language_code | VARCHAR(10) | Y | 翻訳表示言語コード |
| show_original | BOOLEAN | N | 原文表示可否 |
| show_translation | BOOLEAN | N | 翻訳表示可否 |
| active | BOOLEAN | N | 参加中可否 |
| joined_at | DATETIME | N | 参加日時 |
| left_at | DATETIME | Y | 退出日時 |
| created_at | DATETIME | N | 作成日時 |
| updated_at | DATETIME | Y | 更新日時 |
| deleted_at | DATETIME | Y | 削除日時 |

unique制約:

```sql
CREATE UNIQUE INDEX uk_chat_room_member_room_user
ON chat_room_member(chat_room_id, user_id);
```

index候補:

```sql
CREATE INDEX idx_chat_room_member_user_active
ON chat_room_member(user_id, active);

CREATE INDEX idx_chat_room_member_room_active
ON chat_room_member(chat_room_id, active);
```

## 5. chat_message

| カラム | 型 | NULL | 説明 |
|---|---:|---:|---|
| id | BIGINT | N | PK |
| chat_room_id | BIGINT | N | チャットルームID |
| sender_user_id | BIGINT | Y | 送信ユーザーID |
| sender_type | VARCHAR(30) | N | USER, AI |
| message_type | VARCHAR(30) | N | TEXT, SYSTEM |
| content | TEXT | N | 原文内容 |
| status | VARCHAR(30) | N | SENT, DELETED |
| created_at | DATETIME | N | 作成日時 |
| updated_at | DATETIME | Y | 更新日時 |
| deleted_at | DATETIME | Y | 削除日時 |

AIメッセージ拡張を考慮し、`sender_user_id` はnullableにできる。ただし1次USERメッセージでは必須とする。

index候補:

```sql
CREATE INDEX idx_chat_message_room_created_id
ON chat_message(chat_room_id, created_at, id);

CREATE INDEX idx_chat_message_room_id_id
ON chat_message(chat_room_id, id);
```

## 8. 家計簿連携拡張テーブル候補

禁止:

```text
chat_room.account_book_id
```

推奨:

```text
chat_room_account_book_link
```

関係:

```text
chat_room 1 ─ N chat_room_account_book_link N ─ 1 account_book
```

原則:

- チャット権限と家計簿権限は分離する。
- 家計簿アクセスは `account_book_member` 基準で検証する。
- 1つのチャットルームに複数家計簿を連携できる。

---

## 9. メッセージ取得方針

初回取得:

```text
最新100件
```

過去取得:

```text
cursorMessageId基準で以前100件
```

DB取得ソート:

```text
created_at DESC, id DESC
```

FE表示ソート:

```text
created_at ASC, id ASC
```
